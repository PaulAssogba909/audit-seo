## PARTIE 0 — DÉCISIONS ARCHITECTURALES DÉFINITIVES

Avant toute action, ces décisions sont figées et ne changent plus.

**Stack technique**
- Backend : Node.js + Express sur Railway
- Image Docker : `mcr.microsoft.com/playwright/node:latest`
- File de jobs : BullMQ + Redis (plugin Railway)
- DB interne : PostgreSQL (plugin Railway)
- Temps réel : WebSocket (socket.io)
- Rognage : Sharp (Node.js)
- Captures : Playwright (Chromium headless)
- IA : GPT-4o Vision (rognage + extraction + décisions)
- Stockage images : Cloudinary
- Données audit : Airtable

**Principe IA-first pour toutes les captures**
Playwright fait la capture brute → GPT-4o rogne et extrait → Sharp applique le crop → Cloudinary stocke → Airtable reçoit l'URL. Aucun crop en pourcentage fixe dans le code.

**Principe sessions tierces (Ubersuggest, MRM)**
Playwright visible → utilisateur se connecte lui-même → cookies capturés automatiquement → chiffrés AES-256 → stockés en DB → réinjectés à chaque audit.

**Conflict résolu**
`Img_longeur_page` devient deux champs distincts dans Airtable :
- `Img_longeur_page` → feuille "Nb de mots body" du sheet audit
- `Img_longeur_page_plan` → feuille "Longueur de page" du sheet plan d'action

**Ordre d'exécution des captures (stratégique)**
1. Robots.txt + Sitemap + Logo (sans auth, sans risque)
2. AmIResponsive (sans auth)
3. SSL Labs (sans auth, lent → lancer tôt)
4. PageSpeed Mobile + Desktop (Playwright + IA)
5. Google Sheets audit (OAuth Google)
6. Google Sheets plan d'action (OAuth Google)
7. Google Search Console (OAuth Google)
8. My Ranking Metrics (session utilisateur)
9. Ubersuggest (session utilisateur)
10. Semrush (anti-bot risqué → en avant-dernier)
11. Ahrefs (anti-bot le plus risqué → en dernier)

## PARTIE 2 — BASE DE DONNÉES INTERNE

### ACTION 2.1 — Créer le schéma PostgreSQL

**Table users**
```
id (UUID, PK)
email (TEXT, UNIQUE)
password_hash (TEXT)
google_refresh_token (TEXT, chiffré)
created_at (TIMESTAMP)
```

**Table audits**
```
id (UUID, PK)
user_id (UUID, FK users)
nom_site (TEXT)
url_site (TEXT)
sheet_audit_url (TEXT)
sheet_plan_url (TEXT)
mrm_report_url (TEXT)
airtable_record_id (TEXT)     ← ID de l'enregistrement Airtable créé
statut_global (TEXT)          ← EN_ATTENTE / EN_COURS / TERMINE / TERMINE_ERREURS
created_at (TIMESTAMP)
```

**Table audit_steps**
```
id (UUID, PK)
audit_id (UUID, FK audits)
step_key (TEXT)               ← ex: "robots_txt", "psi_mobile"
statut (TEXT)                 ← EN_ATTENTE / EN_COURS / IA_EN_COURS / SUCCES / SKIP / ERREUR / INTERVENTION
attempts (INTEGER, default 0)
error_message (TEXT)
output_cloudinary_url (TEXT)
output_value (TEXT)           ← pour les valeurs texte (scores PSI, URL robots, etc.)
started_at (TIMESTAMP)
ended_at (TIMESTAMP)
```

**Table audit_events**
```
id (UUID, PK)
audit_id (UUID, FK)
step_key (TEXT)
type (TEXT)                   ← LOG / WARN / ERROR / AI_DECISION
message (TEXT)
timestamp (TIMESTAMP)
```

**Table user_sessions**
```
id (UUID, PK)
user_id (UUID, FK)
service (TEXT)                ← "ubersuggest" / "mrm"
encrypted_cookies (TEXT)      ← JSON chiffré AES-256
expires_at (TIMESTAMP)
created_at (TIMESTAMP)
```

### ACTION 2.2 — Créer les index PostgreSQL

- Index sur `audit_events.audit_id`

## PARTIE 7 — MOTEUR DE JOBS (CŒUR DU PROJET)

### ACTION 7.1 — Configuration BullMQ

**Queue "audit-jobs"**
- Concurrence : 2 jobs simultanés maximum (limité par la RAM Railway)
- Retry automatique : 0 (les retries sont gérés au niveau des steps, pas des jobs)
- Timeout job global : 90 minutes

**Queue "step-retry"**
- Pour les relances individuelles d'un step depuis l'UI
- Priorité haute

### ACTION 7.2 — Structure du Worker principal

Le worker reçoit `{ auditId, userId }` et exécute les steps dans l'ordre défini.

Pour chaque step :
```
1. Vérifier les préconditions (token valide, session valide, sheet accessible)
   → Si précondition échoue : SKIP avec message explicatif → step suivant

2. Mettre à jour DB : statut = EN_COURS + started_at
3. Émettre WebSocket : { stepKey, statut: "EN_COURS" }

4. Exécuter l'action Playwright/API (avec timeout strict)
   → Succès partiel ou total → aller en 5
   → Exception → aller en 8 (boucle IA de décision)

5. Si capture : mettre DB statut = IA_EN_COURS
   Émettre WebSocket : { stepKey, statut: "IA_EN_COURS" }

6. Envoyer capture brute à GPT-4o (rognage + extraction)
   → Réponse valide → aller en 7
   → Réponse invalide → re-prompt (max 3 fois) → si toujours invalide → aller en 8

7. Appliquer crop avec Sharp
   Upload Cloudinary → récupérer URL
   Si DRY_RUN=false : écrire dans Airtable (URL + valeurs texte)
   Mettre DB : statut = SUCCES + output_cloudinary_url + ended_at
   Émettre WebSocket : { stepKey, statut: "SUCCES", imageUrl }
   → Step suivant

8. BOUCLE IA DE DÉCISION (max 3 tentatives par step)
   Prendre screenshot état actuel de la page
   Envoyer à GPT-4o : screenshot + logs d'erreur + description du step + historique tentatives
   GPT-4o répond avec une action parmi la liste autorisée :
   
   CLOSE_POPUP → fermer popup visible
   SCROLL_TO_TEXT:"texte" → scroller jusqu'au texte
   CLICK_ELEMENT:"selector" → cliquer sur un élément
   WAIT_SECONDS:N → attendre N secondes
   RETRY → réessayer l'action depuis le début
   SKIP:"raison" → ignorer ce step avec message
   REQUEST_HUMAN → passer en statut INTERVENTION
   
   Exécuter l'action décidée → retourner en 4 (tentative suivante)
   
   Après 3 tentatives échouées :
   → Mettre DB : statut = ERREUR + error_message
   → Émettre WebSocket : { stepKey, statut: "ERREUR", message }
   → Continuer avec le step suivant (l'échec d'un step n'arrête pas l'audit)
```

### ACTION 7.3 — Fin d'audit
- Tous les steps terminés (SUCCES + SKIP + ERREUR)
- Calculer le statut global :
  - 0 erreurs → "Terminé"
  - Au moins 1 erreur → "Terminé avec erreurs"
- Mettre à jour DB + Airtable statut
- Émettre WebSocket : audit terminé
- (optionnel) Envoyer notification email

## PARTIE 10 — GESTION GLOBALE DES ERREURS

### ACTION 10.1 — Site inaccessible
- Avant de lancer le worker : faire un GET axios sur l'URL du site (timeout 15s)
- Si timeout ou erreur réseau → annuler l'audit immédiatement → notifier l'utilisateur
- Si redirection vers page de maintenance → continuer mais noter dans les events

### ACTION 10.2 — Nettoyage systématique
- Après chaque upload Cloudinary : supprimer le fichier temporaire du disque Railway
- Si le step échoue avant l'upload : supprimer quand même le fichier temporaire
- Fermer proprement chaque contexte Playwright après chaque step (pas une instance globale)
- Le browser pool gère le recyclage des instances

### ACTION 10.3 — Monitoring des coûts IA
- Logger chaque appel GPT-4o avec : step_key, tokens_input, tokens_output, coût estimé
- Si le coût d'un audit dépasse un seuil configurable → log d'alerte (ne pas bloquer l'audit)

### ACTION 10.4 — Mode dry-run
- Variable `DRY_RUN=true` → toutes les étapes s'exécutent normalement sauf :
  - Pas d'écriture dans Airtable
  - Les images sont quand même uploadées sur Cloudinary dans un dossier "debug"
  - Tous les logs sont conservés
- Utile pendant le développement pour tester sans polluer Airtable
